# ADR 0068: Phase 2a.19 — Daily Brief On-Device Composer

## Status

Accepted

## Context

ADR 0067 shipped the `daily_brief` action type as part of the §9C
coverage pass, but the tool returned only a metadata envelope. The
shell rendered runtime + section labels, with a note that "the
vernacular text would be generated by the Tier 4 SLM once installed."

That was honest but underwhelming. §9C vignette 16b describes a
specific user moment: *"At 6:30 AM Priya's phone reads her a
90-second brief in Tamil"*. Without an actual brief body, the
investor demo of vignette 16b is just a card with no story.

Phase 2a.19 closes the gap with a template-based renderer that
consumes a structured signals payload (recent activity, mesh
earnings, expiring consents, open §9A flags) and emits a short
vernacular text in the user's locale. The Tier 4 generative SLM
will replace the template once it lands; the signals contract
stays the same.

## Decision

### New artifact — `src/phase1/daily-brief.mjs`

Two pure functions:

- **`gatherDailyBriefSignals(store, identityId, { horizonHours, at })`**
  reads from the `BosStore`:
  - last 5 orchestrations from this identity in the horizon
  - mesh contribution events summed to `{ earnedPaise, tokens, bytes, eventCount }`
  - up to 3 consents expiring in the next 7 days
  - count of open §9A flag reports against the identity

  Returns a `signals` object with `protocolVersion`,
  `horizonHours`, `asOf`, and the four sections.

- **`renderDailyBrief({ signals, locale, displayName })`** emits a
  short multi-line text in the requested locale (en-IN, hi-IN,
  hi-Latn-IN, mr-IN, bho-IN, ta-IN, bn-IN). Each section has a
  populated and an empty-state template, so the brief always reads
  naturally even on a quiet day. The footer carries the §7e
  on-device framing in every locale.

### API integration — `src/phase0/api.mjs`

The `POST /api/orchestrations` handler detects
`actionType === 'daily_brief'` and, before dispatching, calls
`gatherDailyBriefSignals` against the actor identity and threads
the result into `request.metadata.signals`. It also injects
`subjectDisplayName` so the greeting can address the user by name.

This is the server stand-in for Phase 2b on-device gather. §15
binding holds because the signals only flow back to the same client
in the orchestration response — they never persist outside the
user's profile boundary.

### Tool adapter — `src/phase1/tools.mjs`

`daily_brief_compose` now calls `renderDailyBrief` when
`metadata.signals` is present and embeds the result on the receipt
as `brief: { protocolVersion, locale, text, lineCount,
sectionsPopulated, renderer: 'template_v0', rendererNote, rawPiiReturned: false }`.

The `rendererNote` explicitly says *"Template-based renderer. Tier
4 generative SLM (Sarvam-1 q4 / Gemma 2 q4) will replace this with
fluent prose once installed."* — the §17 honesty board literally
labels the moment.

### Shell rendering — `public/shell/app.js` + `styles.css`

The result card for `daily_brief` now leads with the rendered text
in a `<pre class="daily-brief-body">` block (vernacular text, with
the right `lang` attribute for screen readers and TTS), followed by
the existing metadata grid (now showing `renderer` and `sectionsPopulated`).

Service worker bumped `v14 → v15`.

## §15 bindings preserved

| Binding | Resolution |
|---|---|
| Pointer, not payload | Signals are aggregates (counts, paise totals, purpose labels). No raw transactions, no consent ciphertexts, no flag report bodies. |
| Identity is the person, not the device | The gather reads from the active profile's records; same profile on a different device produces the same brief. |
| Never sell user data | `rawPiiReturned: false` on every receipt. No telemetry on which sections fired. |
| Monetize businesses, never citizens | `revenueLine: 'none — citizen-facing (§15 binding)'` carried through from ADR 0067. |

## Tests

`tests/node/daily-brief.test.mjs` — 8 focused tests:

1. empty signal shape for a fresh identity
2. mesh paise totals fold across events
3. horizon window filters out-of-range events
4. every supported locale produces non-empty text + correct sections
5. empty-state lines fire when nothing populated
6. open §9A flag warning surfaces
7. §7e footer present in every locale
8. missing signals throws

Full suite: **228 / 228 green** (was 220; +8 new). SW cache to v15.

## Consequences

- The §9C vignette 16b demo now reads as a real moment. The user
  taps *"Give me my morning brief"* (in any of seven locales), gets
  back a short vernacular text addressed by name, with mesh
  earnings, recent action labels, consent expiries, and §9A flag
  warnings rolled into a few lines.
- The Tier 4 SLM swap is now a 50-line refactor inside
  `dailyBriefCompose`, not a multi-day integration. The signals
  contract is fixed; the renderer is the only variable.
- Template-based generation is honestly labeled. Investors see the
  brief body AND the `renderer: 'template_v0'` row + the explicit
  note. No "demo magic" claims.

## Future polish

- Wire the warmed-up Tier 3 embedding SLM (transformers.js +
  paraphrase-multilingual-MiniLM-L12-v2) to *rank* recent activity
  by semantic relevance to the user's stated focus (read from the
  prompt, e.g. *"focus on work today"* → boost work-tagged
  activity in the `recent` section). The renderer stays template-
  based; the SLM picks which signals to surface.
- Add a `tts` step: once the brief renders, push the text to the
  shell's existing `speechSynthesis` integration so the brief reads
  itself aloud. The Phase 2a.6 TTS runtime already exists.
- Add a *"compose brief at 6:30 AM"* schedule that fires a push
  notification with the brief preview. Phase 2a.4 push
  infrastructure is in place.
- Wire `health_record_read` and `service_booking` receipts as
  named sections so the brief can include *"your HbA1c is due in
  two weeks"* and *"your hotel in Munnar is next Friday"*. These
  are richer than the current `summary` line.
